# Personal Finance Tracker - Practical Assessment Documentation

## 1. Entity Relationship Diagram (ERD) - 6 Tables (3 Marks)

### Tables and Relationships:

1. **User** - Stores user account information
2. **Location** - Stores hierarchical location data (Province → District → Sector)
3. **Account** - Stores user's financial accounts (bank accounts, mobile money)
4. **Transaction** - Stores financial transactions
5. **Category** - Stores transaction categories
6. **Budget** - Stores user budget information
7. **UserProfile** - Stores additional user profile information (NEW)
8. **user_favorite_categories** - Join table for Many-to-Many relationship (NEW)

### Relationship Logic:

- **User ↔ Location**: Many-to-One (Many users can live in one location)
- **User ↔ Account**: One-to-Many (One user can have multiple accounts)
- **User ↔ Transaction**: One-to-Many (One user can have multiple transactions)
- **User ↔ UserProfile**: One-to-One (Each user has one profile)
- **User ↔ Category**: Many-to-Many (Users can favorite multiple categories, categories can be favorited by multiple users)
- **Account ↔ Transaction**: One-to-Many (One account can have multiple transactions)
- **Category ↔ Transaction**: Many-to-One (Many transactions belong to one category)
- **Location ↔ Location**: Self-referencing (Parent-child hierarchy for provinces, districts, sectors)

## 2. Implementation of Saving Location (2 Marks)

### Entity: Location.java
```java
@Entity
public class Location {
    @Id
    @GeneratedValue(generator = "UUID")
    @GenericGenerator(name = "UUID", strategy = "org.hibernate.id.UUIDGenerator")
    private UUID id;
    
    private String code;
    private String name;
    private String type; // PROVINCE, DISTRICT, SECTOR
    
    @ManyToOne
    @JoinColumn(name = "parent_location_id")
    private Location parentLocation; // Self-referencing for hierarchy
}
```

### How Data is Stored:
1. **UUID Primary Key**: Each location has a unique UUID identifier auto-generated by Hibernate
2. **Hierarchical Structure**: Uses self-referencing ManyToOne relationship
   - Province has `parentLocation = null`
   - District has `parentLocation = Province`
   - Sector has `parentLocation = District`
3. **Foreign Key**: `parent_location_id` column stores the parent's UUID

### Example:
```
Province: Kigali (parent_location_id = NULL)
  ├─ District: Gasabo (parent_location_id = Kigali's UUID)
  │   └─ Sector: Kimironko (parent_location_id = Gasabo's UUID)
```

### Endpoint:
- **POST** `/api/locations` - Creates new location
- **PUT** `/api/locations/{id}` - Updates existing location

## 3. Implementation of Sorting & Pagination (5 Marks)

### A. Pagination Implementation

**Repository Layer** (TransactionRepository.java):
```java
public interface TransactionRepository extends JpaRepository<Transaction, Long> {
    Page<Transaction> findByAccountId(Long accountId, Pageable pageable);
    Page<Transaction> findByUserId(Long userId, Pageable pageable);
}
```

**Service Layer** (TransactionService.java):
```java
public Page<Transaction> getAllTransactionsPaginated(Pageable pageable) {
    return transactionRepository.findAll(pageable);
}
```

**Controller Layer** (TransactionController.java):
```java
@GetMapping("/paginated")
public Page<Transaction> getAllTransactionsPaginated(
        @RequestParam(defaultValue = "0") int page,
        @RequestParam(defaultValue = "10") int size,
        @RequestParam(defaultValue = "id") String sortBy,
        @RequestParam(defaultValue = "DESC") String sortDirection) {
    Sort sort = sortDirection.equalsIgnoreCase("ASC") ? 
                Sort.by(sortBy).ascending() : Sort.by(sortBy).descending();
    Pageable pageable = PageRequest.of(page, size, sort);
    return transactionService.getAllTransactionsPaginated(pageable);
}
```

### B. How Pagination Works:

1. **Pageable Interface**: Spring Data JPA's abstraction for pagination
2. **PageRequest.of(page, size, sort)**: Creates pagination request
   - `page`: Page number (0-indexed)
   - `size`: Number of records per page
   - `sort`: Sorting criteria
3. **Page<T> Return Type**: Contains:
   - `content`: List of entities for current page
   - `totalElements`: Total records in database
   - `totalPages`: Total number of pages
   - `number`: Current page number
   - `size`: Page size

### C. Performance Benefits:

1. **Reduced Memory Usage**: Only loads requested page into memory, not entire dataset
2. **Faster Query Execution**: Database uses LIMIT and OFFSET clauses
3. **Better User Experience**: Quick response times even with large datasets
4. **Network Efficiency**: Transfers less data over network

### D. Sorting Implementation:

**Sort.by(field).ascending()** or **Sort.by(field).descending()**
- Sorts results by specified field
- Can sort by any entity field (id, transactionDate, amount, etc.)
- SQL: Translates to `ORDER BY field ASC/DESC`

### Example API Calls:
```
GET /api/transactions/paginated?page=0&size=10&sortBy=transactionDate&sortDirection=DESC
GET /api/transactions/account/1/paginated?page=1&size=20&sortBy=amount&sortDirection=ASC
```

## 4. Implementation of Many-to-Many Relationship (3 Marks)

### Entities:

**User.java**:
```java
@ManyToMany
@JoinTable(
    name = "user_favorite_categories",
    joinColumns = @JoinColumn(name = "user_id"),
    inverseJoinColumns = @JoinColumn(name = "category_id")
)
private Set<Category> favoriteCategories = new HashSet<>();
```

**Category.java**:
```java
@ManyToMany(mappedBy = "favoriteCategories")
private Set<User> users = new HashSet<>();
```

### Join Table Explanation:

**Table Name**: `user_favorite_categories`

**Columns**:
- `user_id` (Foreign Key → users.id)
- `category_id` (Foreign Key → category.id)

**How It Works**:
1. **@JoinTable**: Specifies the join table name and column mappings
2. **joinColumns**: Defines the foreign key column for the owning side (User)
3. **inverseJoinColumns**: Defines the foreign key column for the inverse side (Category)
4. **mappedBy**: On Category side, indicates User is the owner of the relationship

### Example Data:
```
user_favorite_categories table:
| user_id | category_id |
|---------|-------------|
| 1       | 5           | (User 1 favorites Category 5 - Food & Groceries)
| 1       | 8           | (User 1 favorites Category 8 - Transport)
| 2       | 5           | (User 2 favorites Category 5 - Food & Groceries)
```

## 5. Implementation of One-to-Many Relationship (2 Marks)

### Example: User → Accounts

**User.java** (One side):
```java
@OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
private List<Account> accounts;
```

**Account.java** (Many side):
```java
@ManyToOne
@JoinColumn(name = "user_id")
private User user;
```

### Relationship Mapping Explanation:

1. **@OneToMany**: Indicates one User can have many Accounts
2. **mappedBy = "user"**: Specifies that Account entity owns the relationship (has the foreign key)
3. **cascade = CascadeType.ALL**: When User is deleted, all associated Accounts are deleted
4. **@ManyToOne**: On Account side, indicates many Accounts belong to one User
5. **@JoinColumn(name = "user_id")**: Creates foreign key column in account table

### Foreign Key Usage:

**Database Structure**:
```
users table:
| id | username | email | ... |
|----|----------|-------|-----|
| 1  | john     | ...   | ... |

account table:
| id | name          | balance | user_id (FK) |
|----|---------------|---------|--------------|
| 1  | Bank of Kigali| 50000   | 1            |
| 2  | MTN Mobile    | 25000   | 1            |
```

The `user_id` column in account table references `id` in users table, establishing the relationship.

### Entities:

**User.java**:

```java
@OneToOne(mappedBy = "user", cascade = CascadeType.ALL)
```


### How Entities are Connected:

1. **@OneToOne**: Indicates one User has exactly one UserProfile
2. **@JoinColumn(name = "user_id", unique = true)**: 
   - Creates `user_id` foreign key column in user_profile table
   - `unique = true` ensures one profile per user
3. **mappedBy = "user"**: On User side, indicates UserProfile owns the relationship

### Database Structure:
```
users table:
| id | username | email | ... |
|----|----------|-------|-----|
| 1  | john     | ...   | ... |

user_profile table:
| id | user_id (FK, UNIQUE) | bio       | date_of_birth | occupation |
|----|----------------------|-----------|---------------|------------|
| 1  | 1                    | "Hello"   | 1990-05-15    | Engineer   |
```

### Key Points:
- Each User can have only ONE UserProfile
- Each UserProfile belongs to only ONE User
- Foreign key has UNIQUE constraint to enforce one-to-one relationship

## 7. Implementation of existBy() Method (2 Marks)

### Repository Layer:

**UserRepository.java**:
```java
public interface UserRepository extends JpaRepository<User, Long> {
    boolean existsByEmail(String email);
    boolean existsByUsername(String username);
}
```

**UserProfileRepository.java**:
```java
public interface UserProfileRepository extends JpaRepository<UserProfile, Long> {
    boolean existsByUserId(Long userId);
}
```

### How Existence Checking Works:

1. **Spring Data JPA Query Derivation**: 
   - Method name `existsByEmail` is automatically parsed
   - Spring generates SQL: `SELECT COUNT(*) > 0 FROM users WHERE email = ?`

2. **Return Type**: 
   - Returns `boolean` (true if exists, false if not)
   - More efficient than fetching entire entity

3. **Performance**:
   - Uses COUNT query instead of SELECT
   - Database only returns true/false, not full entity data
   - Faster than `findByEmail() != null` check

### Service Layer:

**UserService.java**:
```java
public boolean existsByEmail(String email) {
    return userRepository.existsByEmail(email);
}

public boolean existsByUsername(String username) {
    return userRepository.existsByUsername(username);
}
```

### Controller Layer:

## 8. Retrieve Users by Location (4 Marks)

### Repository Layer:

**UserRepository.java**:
```java
// Find users by location code
@Query("SELECT u FROM User u WHERE u.location.code = :code")
List<User> findByLocationCode(@Param("code") String code);

// Find users by location name
@Query("SELECT u FROM User u WHERE u.location.name = :name")
List<User> findByLocationName(@Param("name") String name);
```

### Query Logic Explanation:

1. **@Query Annotation**: Defines custom JPQL (Java Persistence Query Language) query
2. **JPQL Syntax**: 
   - `SELECT u FROM User u`: Select User entities (alias 'u')
   - `WHERE u.location.code = :code`: Filter by location's code field
   - Navigates relationship using dot notation (u.location.code)
3. **@Param Annotation**: Binds method parameter to query parameter
4. **Return Type**: List<User> - Returns all matching users

### Service Layer:

**UserService.java**:
```java
public List<User> getUsersByLocationCode(String code) {
    return userRepository.findByLocationCode(code);
}

public List<User> getUsersByLocationName(String name) {
    return userRepository.findByLocationName(name);
}
```

### Controller Layer:

**UserController.java**:
```java
@GetMapping("/location/code/{code}")
public List<User> getUsersByLocationCode(@PathVariable String code) {
    return userService.getUsersByLocationCode(code);
}

@GetMapping("/location/name/{name}")
public List<User> getUsersByLocationName(@PathVariable String name) {
    return userService.getUsersByLocationName(name);
}
```

### How It Works:

1. **Client Request**: `GET /api/users/location/code/KGL`
2. **Controller**: Receives code parameter "KGL"
3. **Service**: Calls repository method
4. **Repository**: Executes JPQL query
5. **Database**: Performs JOIN between users and location tables
   ```sql
   SELECT u.* FROM users u 
   INNER JOIN location l ON u.location_id = l.id 
   WHERE l.code = 'KGL'
   ```
6. **Response**: Returns list of all users in Kigali province

### Example API Calls:
```
GET /api/users/location/code/KGL → Returns all users in Kigali
GET /api/users/location/name/Gasabo → Returns all users in Gasabo district
```

### Use Cases:
- Find all users in a specific province for regional analysis
- Get users in a district for targeted notifications
- Filter users by location for location-based features
